# 2026-05-16

## Followed up on PR #84 (`ViewOption` → `BasesAllOptions`)

PR #84 (merged) renamed the imported view-option type to match the current
Obsidian API. Two cleanups applied on top:

- Bumped `obsidian` in `bun.lock` from `1.10.3` to `1.12.3` (`package.json`
  stays on `"latest"`). `BasesAllOptions` was first added in `obsidian@1.12.0`,
  so without the lockfile bump `bun run tsc` would have failed against the
  previously resolved version.
- Bumped `manifest.json` `minAppVersion` from `1.10.0` to `1.12.0` to match the
  Obsidian version where `BasesAllOptions` is available.
- Removed the misleading "Obsidian v1.10.3 seemingly replaced ViewOption with
  BasesAllOptions" comment from both option files — the rename actually
  happened in `1.12.0`, and the comment violates the project's default-no-
  comments rule anyway.

## Note on the original motivation

PR #84's description attributes broken heatmaps and chart state to this type
rename. That can't be the cause — TypeScript types are erased at compile time
and the data shapes of `ViewOption`/`DropdownOption` (1.10.x) and
`BasesAllOptions`/`BasesDropdownOption` (1.12.x) are structurally identical
(same `type` discriminators, same fields). Whatever runtime issue the reporter
hit is unrelated and still needs investigation if it recurs.
