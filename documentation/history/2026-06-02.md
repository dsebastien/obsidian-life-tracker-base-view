## Option to hide the Life Tracker header controls (#88)

The Life Tracker view always rendered the time frame / overlay / reset / columns
toolbar above the grid. Users who lock those settings in a view want a more
compact, distraction-free layout.

### What changed

- New per-view toggle `hideControlBar` in the Layout group of the view options
  panel (label: "Hide header controls", default `false`).
- `LifeTrackerView` reads the toggle and skips `createGridControls` when set,
  so the toolbar (time frame dropdown, overlay button, reset-order button,
  columns +/-) is not rendered.
- The grid layout, drag-and-drop, time frame filtering and all other behavior
  are unchanged â€” the toggle only controls the toolbar's presence.
- `canDoIncrementalUpdate` now invalidates the fast path when the toggle
  changes, so flipping it in the settings panel immediately shows/hides the
  toolbar without needing another structural change.

### Files

- `src/app/view/view-options.ts` â€” added `hideControlBar` toggle.
- `src/app/view/life-tracker-view.ts` â€” `previousHideControlBar` tracking,
  conditional `createGridControls`, fast-path invalidation.
- `docs/configuration.md` â€” documented the new option.

## Fixed: heatmap 0 values colored on a 0-based scale (issue #87)

Reporter (#87) had a blue heatmap configured with scale 0-5 and entries
of 0 or 1. After the 2.8.1 release, cells with value 0 and value 1 both
rendered as the first colored level instead of 0 rendering empty (gray).
The legend at the bottom shows an empty slot for 0, so users expect 0 to
render as absence.

### Root cause

The fix for #76 made every present value at least level 1 so that cells
at min (e.g. value=2025 in a year-range heatmap with min=2025, max=2026)
stayed visible. That overshot: when the scale floor is `0`, value `0`
means "no occurrences" and should render empty, not level 1.

### What changed

- `src/utils/color.utils.ts` â€” added an explicit `min === 0 && value === 0`
  branch that returns level 0 before the at-min visibility rule. Updated
  the function doc to call out both issue references so the next person
  doesn't re-collide the two cases.
- `src/utils/color.utils.spec.ts` â€” tests for the 0-on-0-scale case
  (issue #87 regression), the equal-min-max-at-zero edge case, and a
  retained negative-range test confirming `value=0, min=-100` still maps
  to a colored level (0 with non-zero min is mid-scale, not absence).

### Why `min === 0` and not `value === 0` alone

The #76 case had min=2025 (auto-computed from year data). `value === 0`
alone would have re-broken that scenario if a similar auto-computed range
included 0. Gating on `min === 0` is the conservative cut: it covers the
intended use case (user explicitly configures a counter-style scale
starting at 0) without re-introducing the year-range regression.
