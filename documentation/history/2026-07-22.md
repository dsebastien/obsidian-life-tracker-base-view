# 2026-07-22

## Decimal values in number property inputs (issue #120, PR #119)

Number-type property definitions (weight, mood, body-fat %, ‚Ä¶) silently rejected
fractional values in the Capture Today / Capture Properties modal. Typing `82.75`
and pressing Enter cleared the field.

### Root cause

`src/app/components/editing/number-editor.ts` rendered its text inputs with
`inputmode: 'numeric'` and `pattern: '[0-9]*'`. The HTML5 `pattern` only matched
integers, so the browser rejected decimals on blur/submit ‚Äî even though the
keystroke blocker (`isValidNumberKeystroke`, regex `/^-?\d*\.?\d*$/`) and paste
sanitizer already accepted decimals and negatives. The slider used `step: '1'`,
snapping nudges to integers.

### What changed

- `src/app/components/editing/editing.constants.ts` ‚Äî added exported
  `NUMBER_INPUT_ATTRS` (`inputmode: 'decimal'`, `pattern: '-?[0-9]*\.?[0-9]*'`)
  and `NUMBER_SLIDER_STEP` (`'0.01'`). Centralized so both render paths stay in
  sync and the pattern is unit-testable.
- `src/app/components/editing/number-editor.ts` ‚Äî both render paths
  (slider+input and plain input) now spread `NUMBER_INPUT_ATTRS` and use
  `NUMBER_SLIDER_STEP`, replacing the duplicated inline `numeric`/`[0-9]*`/`'1'`.
- `src/app/components/editing/editing.constants.spec.ts` ‚Äî new spec (7 tests)
  exercising the pattern the way the browser does (anchored whole-string match):
  accepts integers, decimals, negatives, and partial mid-typing input; rejects
  `82.75kg`, `abc`, `1,5`, `1e3`. Also asserts `inputmode` and a fractional
  slider step.

### Decisions

- Chose slider `step: '0.01'` over `'any'`: keeps the thumb snapping to clean
  two-decimal values instead of noisy arbitrary-precision floats. `NumberRange`
  has no integer/decimal flag, so this is a global choice for all ranged number
  properties.
- Took inspiration from community PR #119 but implemented independently
  (deduplicated attrs into shared constants, added test coverage, fixed the
  PR's comment/code mismatch where a comment said `'any'` but code set `'0.01'`).

### Impact

Backwards compatible. No schema change, no data migration; existing integer
values are unaffected and the `pattern` is strictly more permissive. Only
behavioral change: sliders on ranged number properties now allow 0.01
granularity instead of integer-only.

### Verification

- `bun run validate` (tsc + test + lint) clean ‚Äî 564/564 tests pass.
- `bun run build` succeeds.
- Runtime UI (mobile decimal keypad, blur/submit acceptance of `82.75`, slider
  drag landing on fractional values) **requires manual verification in a live
  vault** ‚Äî agents cannot exercise Obsidian's GUI.

## Type safety: typed view-config accessors + chart casts (issue #113)

Eliminated unsafe `config.get(key) as T` casts and the `as unknown as ChartInstance`
double-casts flagged in the full codebase review.

### What changed

- `src/app/view/config-accessors.ts` (new) ‚Äî runtime-narrowing getters
  `getBoolConfig` / `getNumberConfig` / `getStringConfig` / `getEnumConfig(get, key, allowed)`.
  Each checks the stored value's runtime type (`typeof`, `Number.isFinite`, allowed-set
  membership) and returns `undefined` on mismatch, so callers keep `?? fallback` and a
  malformed stored value (e.g. the string `"true"` for a boolean flag) now falls back to
  the default instead of silently passing the cast.
- `config-accessors.spec.ts` (new, 10 tests) ‚Äî covers the `"true"`-string bypass, `NaN`/
  `Infinity`/numeric-string rejection, empty-string handling, and out-of-set enum values.
- Replaced ~30 bare casts across `life-tracker-view.ts`, `visualization-config.helper.ts`,
  and `grid-view.ts`. Each `BasesView` subclass gained a bound `private readonly cfg:
  ConfigGetter` field so call sites read `getBoolConfig(this.cfg, 'x') ?? false`.
- Added `TIME_GRANULARITY_OPTIONS` and `BATCH_FILTER_MODES` value arrays (mirroring the
  existing `TIME_FRAME_OPTIONS`) so `getEnumConfig` can validate against the allowed set.
- `hideNotesWhen` reads now go through `getEnumConfig(..., BATCH_FILTER_MODES)` in both
  `getFilterMode()` and the filter pass ‚Äî consistent typing (was `as BatchFilterMode`
  in one place, the review had noted a stray `as string`).
- `src/app/types/chart/chart-instance.intf.ts` ‚Äî `ChartInstance` is now `type ChartInstance
  = Chart` (Chart.js's real class) instead of a hand-maintained subset interface. All five
  `new Chart(...) as unknown as ChartInstance` double-casts in `chart-initializers.ts` were
  removed: the constructor result is now directly assignable, so the chart config objects
  and all downstream `.data`/`.options`/`.update()` access are fully type-checked.
- `src/app/view/overlay-id.ts` (new) ‚Äî `overlayIdToPropertyId(overlayId)` wrapper centralizes
  the deliberate reuse of an overlay UUID in the `BasesPropertyId` slot (documented in
  Business Rules ‚Üí Overlay charts), replacing three scattered `as BasesPropertyId` casts on
  raw strings in `life-tracker-view.ts`.

### Decisions

- Getters return `T | undefined` rather than taking a fallback argument: keeps the existing
  `?? default` at call sites (minimal diff, defaults stay co-located) and composes with the
  `columnConfig.x ?? getBoolConfig(...) ?? true` per-viz override chains.
- Kept the overlay‚ÜípropertyId brand reuse (a documented business rule) but expressed it once
  through a named, commented wrapper instead of inventing a parallel `OverlayId` type that
  would just push the same cast downstream.

### Impact

Pure type-safety refactor. Only behavioral change: a config value stored with the wrong
runtime type now resolves to its documented default instead of leaking through. No schema or
data migration. Not user-visible; no docs update.

### Verification

- `bun run validate` (tsc + test + lint) clean ‚Äî 574 tests pass. `bun run build` succeeds.
