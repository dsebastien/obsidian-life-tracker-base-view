# 2026-07-22

## Decimal values in number property inputs (issue #120, PR #119)

Number-type property definitions (weight, mood, body-fat %, …) silently rejected
fractional values in the Capture Today / Capture Properties modal. Typing `82.75`
and pressing Enter cleared the field.

### Root cause

`src/app/components/editing/number-editor.ts` rendered its text inputs with
`inputmode: 'numeric'` and `pattern: '[0-9]*'`. The HTML5 `pattern` only matched
integers, so the browser rejected decimals on blur/submit — even though the
keystroke blocker (`isValidNumberKeystroke`, regex `/^-?\d*\.?\d*$/`) and paste
sanitizer already accepted decimals and negatives. The slider used `step: '1'`,
snapping nudges to integers.

### What changed

- `src/app/components/editing/editing.constants.ts` — added exported
  `NUMBER_INPUT_ATTRS` (`inputmode: 'decimal'`, `pattern: '-?[0-9]*\.?[0-9]*'`)
  and `NUMBER_SLIDER_STEP` (`'0.01'`). Centralized so both render paths stay in
  sync and the pattern is unit-testable.
- `src/app/components/editing/number-editor.ts` — both render paths
  (slider+input and plain input) now spread `NUMBER_INPUT_ATTRS` and use
  `NUMBER_SLIDER_STEP`, replacing the duplicated inline `numeric`/`[0-9]*`/`'1'`.
- `src/app/components/editing/editing.constants.spec.ts` — new spec (7 tests)
  exercising the pattern the way the browser does (anchored whole-string match):
  accepts integers, decimals, negatives, and partial mid-typing input; rejects
  `82.75kg`, `abc`, `1,5`, `1e3`. Also asserts `inputmode` and a fractional
  slider step.

### Decisions

- Chose slider `step: '0.01'` over `'any'`: keeps the thumb snapping to clean
  two-decimal values instead of noisy arbitrary-precision floats. `NumberRange`
  has no integer/decimal flag, so this is a global choice for all ranged number
  properties.
- Took inspiration from community PR #119 but implemented independently
  (deduplicated attrs into shared constants, added test coverage, fixed the
  PR's comment/code mismatch where a comment said `'any'` but code set `'0.01'`).

### Impact

Backwards compatible. No schema change, no data migration; existing integer
values are unaffected and the `pattern` is strictly more permissive. Only
behavioral change: sliders on ranged number properties now allow 0.01
granularity instead of integer-only.

### Verification

- `bun run validate` (tsc + test + lint) clean — 564/564 tests pass.
- `bun run build` succeeds.
- Runtime UI (mobile decimal keypad, blur/submit acceptance of `82.75`, slider
  drag landing on fractional values) **requires manual verification in a live
  vault** — agents cannot exercise Obsidian's GUI.

## Type safety: typed view-config accessors + chart casts (issue #113)

Eliminated unsafe `config.get(key) as T` casts and the `as unknown as ChartInstance`
double-casts flagged in the full codebase review.

### What changed

- `src/app/view/config-accessors.ts` (new) — runtime-narrowing getters
  `getBoolConfig` / `getNumberConfig` / `getStringConfig` / `getEnumConfig(get, key, allowed)`.
  Each checks the stored value's runtime type (`typeof`, `Number.isFinite`, allowed-set
  membership) and returns `undefined` on mismatch, so callers keep `?? fallback` and a
  malformed stored value (e.g. the string `"true"` for a boolean flag) now falls back to
  the default instead of silently passing the cast.
- `config-accessors.spec.ts` (new, 10 tests) — covers the `"true"`-string bypass, `NaN`/
  `Infinity`/numeric-string rejection, empty-string handling, and out-of-set enum values.
- Replaced ~30 bare casts across `life-tracker-view.ts`, `visualization-config.helper.ts`,
  and `grid-view.ts`. Each `BasesView` subclass gained a bound `private readonly cfg:
ConfigGetter` field so call sites read `getBoolConfig(this.cfg, 'x') ?? false`.
- Added `TIME_GRANULARITY_OPTIONS` and `BATCH_FILTER_MODES` value arrays (mirroring the
  existing `TIME_FRAME_OPTIONS`) so `getEnumConfig` can validate against the allowed set.
- `hideNotesWhen` reads now go through `getEnumConfig(..., BATCH_FILTER_MODES)` in both
  `getFilterMode()` and the filter pass — consistent typing (was `as BatchFilterMode`
  in one place, the review had noted a stray `as string`).
- `src/app/types/chart/chart-instance.intf.ts` — `ChartInstance` is now `type ChartInstance
= Chart` (Chart.js's real class) instead of a hand-maintained subset interface. All five
  `new Chart(...) as unknown as ChartInstance` double-casts in `chart-initializers.ts` were
  removed: the constructor result is now directly assignable, so the chart config objects
  and all downstream `.data`/`.options`/`.update()` access are fully type-checked.
- `src/app/view/overlay-id.ts` (new) — `overlayIdToPropertyId(overlayId)` wrapper centralizes
  the deliberate reuse of an overlay UUID in the `BasesPropertyId` slot (documented in
  Business Rules → Overlay charts), replacing three scattered `as BasesPropertyId` casts on
  raw strings in `life-tracker-view.ts`.

### Decisions

- Getters return `T | undefined` rather than taking a fallback argument: keeps the existing
  `?? default` at call sites (minimal diff, defaults stay co-located) and composes with the
  `columnConfig.x ?? getBoolConfig(...) ?? true` per-viz override chains.
- Kept the overlay→propertyId brand reuse (a documented business rule) but expressed it once
  through a named, commented wrapper instead of inventing a parallel `OverlayId` type that
  would just push the same cast downstream.

### Impact

Pure type-safety refactor. Only behavioral change: a config value stored with the wrong
runtime type now resolves to its documented default instead of leaking through. No schema or
data migration. Not user-visible; no docs update.

### Verification

- `bun run validate` (tsc + test + lint) clean — 574 tests pass. `bun run build` succeeds.

## Capture modal quick wins (issue #106)

Three of the four listed improvements to the capture flow (the fourth, bulk fill,
was explicitly optional and deferred).

### What changed

- **Start at first unfilled property** — `property-capture-modal.ts` `loadProperties()`
  now opens on the first property that still needs a value instead of always index 0.
  Logic extracted to a pure, tested helper `findFirstUnfilledIndex(definitions, isFilled)`
  in `note-completion.utils.ts`: prefers the first unfilled _required_ property, falls
  back to the first unfilled property of any kind, else index 0.
- **"Today" button** — `date-editor.ts` renders a **Today** button beside the date /
  datetime input (wrapped in `.lt-date-editor-row`). New util `formatNowForInput(includeTime)`
  in `date.utils.ts` produces the local `yyyy-MM-dd` / `yyyy-MM-dd'T'HH:mm` value.
- **Rejected-value feedback** — `list-editor.ts` `addValue()` no longer silently clears a
  value that isn't in `allowedValues`. It shakes the input (`lt-editor-list-input--rejected`
    - `@keyframes lt-shake`), keeps and selects the text, and shows a brief inline
      `role="alert"` message that auto-clears after 1.6s (timer cleaned up in `destroy()`).

### Tests

- `note-completion.utils.spec.ts` (new, 11 tests) — covers `isNoteComplete` (all modes,
  empty-string/null, no-relevant-defs) and `findFirstUnfilledIndex` (required-first,
  skip-filled, non-required fallback, all-filled → 0, empty list → 0).
- `date.utils.spec.ts` — 3 new `formatNowForInput` tests (pattern shape + date/datetime
  prefix consistency).

### Deferred

- **Bulk fill** ("fill all remaining" for boolean/dropdown in batch mode) — listed as
  optional in the issue. Deferred: it's a batch-wide write feature whose UX (scope,
  confirmation, undo) needs live-vault design and can't be self-verified by an agent.

### Docs

- `docs/property-capture.md` — documented first-unfilled start, the Today button, and
  rejected-value feedback.

### Verification

- `bun run validate` (tsc + test + lint) clean. `bun run build` succeeds.
- Runtime UI (Today button click, shake animation, carousel landing position)
  **requires manual verification in a live vault**.

## Configurable number slider step + min/max feedback (issue #107)

### What changed

- **Per-property slider step** — `NumberRange` gains an optional `step?: number`.
  `number-editor.ts` uses it for the slider (`step = numberRange.step` when set and
  positive, else the fractional `NUMBER_SLIDER_STEP` default from the #120 decimal fix).
  Set `1` for whole-number scores, `0.5` for half points, or leave empty for decimals.
- **Settings UI** — `renderNumberConstraints` (settings-tab.ts) adds a **Step** input
  beside Min/Max and preserves `step` across min/max edits (previously each min/max change
  rebuilt the range object and would have dropped it).
- **Min/max inference feedback** — entering only one bound still infers the other
  (`max = min + 100`, or `min = 0`), but now the inferred value is written back into the
  sibling input and announced with a `Notice`, instead of happening silently.
- Inference logic extracted to a pure, tested `computeNumberRange(changed, entered, existing)`
  in `settings/number-range.utils.ts`.

### Tests

- `number-range.utils.spec.ts` (new, 7 tests) — both-bounds, min-only/max-only inference,
  clearing bounds, and step preservation (including on the inferred path).

### Docs

- `docs/property-capture.md` — documented min/max/step and the inference feedback.

### Verification

- `bun run validate` (tsc + test + lint) clean — 594 tests. `bun run build` succeeds.
- Runtime UI (step affecting slider granularity, the inference Notice) **requires manual
  verification in a live vault**.

## Chart tooltip polish (issue #103)

All in the chart initializers / bubble aggregation.

### What changed

- **Integer-aware cartesian tooltips** — line/bar/area tooltips previously always
  `value.toFixed(2)`, so habit counts showed as `3.00`. New
  `chart-format.utils.ts` (`areAllValuesIntegers`, `formatMetricValue`): when every
  plotted value is a whole number the tooltip drops decimals (via `toLocaleString`,
  which also adds thousands separators); otherwise up to two decimals.
- **Exact bubble counts** — `BubblePoint` gains a `count` field, set to the true number
  of entries in `aggregateForBubbleChart`. The bubble tooltip reads it directly
  (`"3 entries"`) instead of reverse-engineering a lossy `~count` from the radius.
- **Configurable pie/doughnut legend position** — was hardcoded `'right'`. Added a
  `chartLegendPosition` view option (top/right/bottom/left, default right) →
  `ChartConfig.legendPosition` → `initPieChart`. New `ChartLegendPosition` type +
  `CHART_LEGEND_POSITIONS` validated through the typed `getEnumConfig`.

### Tests

- `chart-format.utils.spec.ts` (new, 6 tests) — integer detection (nulls, negatives,
  fractionals) and formatting (integer drop, ≤2 decimals, rounding).

### Docs

- `docs/configuration.md` — Legend position view option.
- `docs/visualizations.md` — bubble exact-entry tooltip + integer tooltip note.

### Verification

- `bun run validate` (tsc + test + lint) clean — 600 tests. `bun run build` succeeds.
- Tooltip appearance and legend placement **require manual verification in a live vault**.

## Scatter chart: real time axis + safe min/max (issue #97)

### What changed

- **Real date X axis** — `aggregateForScatterChart` now stores the entry's real
  timestamp (`date.getTime()`) as the point's `x` instead of a 0–100% of the date
  range. `initScatterChart` uses a linear X scale titled "Date" with a
  `ticks.callback` that formats each tick timestamp via date-fns (`MMM d, yyyy`),
  and the tooltip shows the actual date (`Jan 3, 2025: 7`) rather than `Time: 42.3%`.
  No date-adapter dependency added — a linear scale with a formatting callback keeps
  the plugin light while giving true temporal spacing.
- **Stack-safe min/max** — new `minOf` / `maxOf` in `utils/number.utils.ts` fold with a
  loop instead of `Math.min(...arr)` / `Math.max(...arr)`, which overflow the call stack
  on very large arrays. The bubble aggregation's date-range and max-count reductions now
  use them; the scatter path no longer needs a range at all.

### Tests

- `number.utils.spec.ts` (new, 4 tests) — extremes, negatives, single element, empty →
  `undefined`, and a 500k-element array that would blow the stack with spread.
- `chart-aggregation.utils.spec.ts` — new assertion that scatter `x` equals the real
  timestamp.

### Docs

- `docs/visualizations.md` — scatter now described as a real date axis.

### Verification

- `bun run validate` (tsc + test + lint) clean — 605 tests. `bun run build` succeeds.
- Axis tick placement / tooltip dates **require manual verification in a live vault**.
