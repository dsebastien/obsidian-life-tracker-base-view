# 2026-05-18

## Fixed: card reordering for overlays (issue #86)

Reporter (#86) couldn't reorder overlay charts. Previously overlays were always
rendered after all individual property cards (per the old "Overlays are
rendered after all individual property visualizations" rule), and the
Obsidian property menu only reorders the property cards themselves — leaving no
way to move an overlay above an individual card.

### What changed

- Drag-and-drop reordering on every card in the Life Tracker grid (property
  cards AND overlay cards). A grip handle (`grip-vertical`) appears in each
  card's top-left corner; pressing and dragging reorders the card.
- Pointer events (not HTML5 drag) so the same code path works on desktop and
  mobile — `isDesktopOnly` is `false` in `manifest.json` and we wanted parity.
- The manual order is persisted per Base view in `manualOrder` under
  `BasesViewConfig`, alongside `columnConfigs` and `overlayConfigs`. Different
  `.base` files and different views of the same `.base` file each get their
  own ordering — no cross-contamination.
- Reconciliation: missing items in the manual order are dropped silently;
  newly added properties/overlays are appended at the end of the natural
  order so they remain discoverable.
- "Reset order" button in the controls bar (only shown when a manual order
  is set) clears `manualOrder` and reverts to Obsidian's property order + overlays
  at the end.

### Subtle bits

- **Unconfigured property cards (no visualization configured yet)** are
  draggable but do NOT get `data-property-id` set. That attribute is what the
  maximize service uses to decide a card is maximizable, and unconfigured
  cards must never be maximized (existing rule in
  `MaximizeStateService.handleMaximizeToggle`). The drag handle uses its own
  `data-card-kind` / `data-card-id` attributes.
- **Hidden-by-overlay properties** stay in the DOM-invisible portion of
  `manualOrder` via the merge in `applyManualOrder`. Without that merge,
  every save would strip them and the next render would re-append them at
  the end, triggering an unnecessary full re-render via the order-signature
  check.
- **`config.set` triggers `onDataUpdated` synchronously** in this codebase
  (existing comment in the time-frame change handler confirms this — that's
  why `isUpdatingGridSettings` exists). Added `isUpdatingManualOrder` and
  pre-write the new signature before `config.set` for double safety.

### Follow-up fix (same day) — broken maximize

Initial implementation regressed the maximize feature: cards stopped going
fullscreen on maximize and stayed inline. Root cause was in `styles.src.css`
— I'd added `.lt-card { position: relative }` in a NEW block placed _after_
`.lt-card--maximized { position: fixed }`. Both selectors have the same
specificity (one class each), so per the CSS cascade the later rule wins. A
maximized card has both classes, so it took `position: relative` instead of
`position: fixed`, killing the fullscreen overlay.

Fix: merged `relative` into the original `.lt-card` block (line 231, before
the maximize rule) and removed the duplicate. Verified in built
`dist/styles.css`: `.lt-card` now declares `position:relative` and
`.lt-card--maximized` correctly overrides it with `position:fixed`.

Same patch also bumped the drag handle's default visibility (was
`opacity-30` on `text-muted`, often invisible on low-contrast themes — now
`opacity-70` with a subtle background, `opacity-100` on hover).

### Second follow-up fix (same day) — drag handles invisible

User reported the drag handles still weren't appearing even after a reload
with the latest build. Root cause: every visualization's `render()` method
calls `this.containerEl.empty()` as its first step (heatmap, chart,
tag-cloud, timeline, base-visualization — `grep -rn 'containerEl.empty'
src/app/components/visualizations/`). The drag handle was being attached
right after the card element was created, then immediately wiped a few
lines later when the visualization rendered.

Fix: moved the `dragController?.attachHandle(cardEl, ...)` calls to AFTER
`visualization.render(dataPoints)` in `renderConfiguredColumn` and AFTER
`visualization.renderChartData(chartData)` in `renderOverlayCard`. The
unconfigured-card path was untouched — `createColumnConfigCard` doesn't
call `.empty()` on the returned card so the original attach-after-create
order works there.

### Third follow-up fix (same day) — unnecessary chart redraws on reorder

User reported the whole view appeared to refresh after every drop, and in
"some other cases". Root cause: `performIncrementalUpdate` always
re-aggregated data points and called `viz.update()` on every visualization
on every `onDataUpdated`, regardless of whether the data actually changed.

The flow that produced the redraws:

1. User drops a card → `applyManualOrder` → `config.set('manualOrder', ...)`.
2. Obsidian fires `onDataUpdated` as a side effect of the config write
   (either synchronously — and the `isUpdatingManualOrder` flag catches it
   — or via a microtask, after the flag is already cleared, so it slips
   through).
3. `canDoIncrementalUpdate` returns true (we pre-set the new order
   signature before `config.set`, so the structure check passes).
4. `performIncrementalUpdate` re-aggregates ALL viz and calls
   `viz.update()` on each — visible to the user as a full-grid refresh.

Fix: `performIncrementalUpdate` now skips visualizations whose data is
still in the cache. `RenderCacheService.startRenderCycle` already clears
the cache for any viz whose entries changed, so a cache hit here means the
data is unchanged and the chart is already correct — no update needed.

Also skipped overlay visualizations in this path (their `propertyId` is a
UUID that doesn't map to a frontmatter property; the per-property
`createDataPoints` call was producing empty data, latently broken). Overlay
structure changes go through the full-render path which handles them
correctly.

Net effect: a drop now triggers DOM reorder + `config.set` + (at worst) a
spurious `onDataUpdated` that does nothing because every viz's data is
cached. The data-changed case (vault save) is unchanged — entries get new
references, cache is invalidated, all stale viz are updated as before.
